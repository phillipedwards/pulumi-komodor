# Komodor Pulumi Provider — Setup & Development Guide

This provider wraps the [Komodor Terraform provider](https://github.com/komodorio/terraform-provider-komodor) via the Pulumi Terraform Bridge, exposing Komodor resources and data sources across all Pulumi-supported languages.

## Prerequisites

Ensure the following tools are installed and present in your `$PATH`:

- [`pulumictl`](https://github.com/pulumi/pulumictl#installation)
- [Go](https://golang.org/dl/) 1.21 or later
- [`golangci-lint`](https://golangci-lint.run/welcome/install/)
- [NodeJS](https://nodejs.org/en/) Active or maintenance version ([Node.js Releases](https://nodejs.org/en/about/previous-releases)).  We recommend using [nvm](https://github.com/nvm-sh/nvm) to manage NodeJS installations.
- [Yarn](https://yarnpkg.com/)
- [TypeScript](https://www.typescriptlang.org/)
- [Python](https://www.python.org/downloads/) (called as `python3`).  For recent versions of MacOS, the system-installed version is fine.
- [.NET](https://dotnet.microsoft.com/download)

Tools are also managed via [`mise`](https://mise.jdx.dev/) — run `mise install` to install pinned versions.

## Repository Structure

- `examples/` — sample code which may optionally be included as integration tests run in CI/CD
- `provider/` — Go code for the provider binary and SDK code generator
  - `provider/cmd/pulumi-tfgen-komodor` — executable that generates the Pulumi resource schema
  - `provider/cmd/pulumi-resource-komodor` — provider plugin executable
  - `provider/resources.go` — Terraform-to-Pulumi mappings and configuration (single source of truth)
- `sdk/` — generated SDKs for each language (do not edit directly)

## Building

```bash
make provider       # Build just the provider binary
make build          # Build provider + all SDKs and install for testing
make generate       # Re-generate schema and SDK source from upstream TF provider
```

Language-specific SDK targets: `generate_dotnet`, `generate_go`, `generate_nodejs`, `generate_python`, `build_dotnet`, `build_go`, `build_nodejs`, `build_python`.

## Running Tests

```bash
make lint_provider      # Run golangci-lint on the provider
make lint_provider.fix  # Auto-fix lint issues
make test_provider      # Run provider unit tests
make test               # Run integration/example tests (requires `make build` first)
```

## Sample Program

1. Create an account with Komodor and generate an API key.

1. Copy the `pulumi-resource-komodor` binary generated by `make provider` into your `$PATH`, e.g.:

    ```bash
    cp bin/pulumi-resource-komodor $GOPATH/bin
    ```

1. Tell Yarn to use your local copy of the SDK:

    ```bash
    make install_nodejs_sdk
    ```

1. Create a new Pulumi program in the `examples/` directory, e.g.:

    ```bash
    mkdir -p examples/my-example/ts
    cd examples/my-example/ts
    pulumi new typescript
    npm install
    yarn link @pulumi/komodor
    ```

1. Set your Komodor API key:

    ```bash
    pulumi config set --secret komodor:apiKey <your-api-key>
    ```

1. Write a minimal program using a Komodor resource (e.g. `komodor.Monitor`) and run `pulumi up`.

1. Verify the resource was created in the Komodor UI, then clean up with `pulumi destroy`.

## End-to-end Testing

Integration tests are Go tests residing in `examples/*_test.go`. Each test executes a program from a subdirectory within `examples/`.

1. Update an example to use a resource from the Komodor provider SDK.
1. Locate the corresponding Go test and remove the `t.Skip(...)` directive.
1. Run the tests:

    ```bash
    make test
    ```

    You can also run each test file separately via test tags:

    ```bash
    (cd examples && go test -v -tags=nodejs)
    ```

1. Repeat for each language and add new examples and tests as needed.

## Configure GitHub Actions CI

This provider uses Pulumi's [ci-mgmt templates](https://github.com/pulumi/ci-mgmt/) to automatically generate CI workflows and other build tooling. Configuration lives in [`.ci-mgmt.yaml`](.ci-mgmt.yaml) and can be regenerated by running `make ci-mgmt`.

The `.github/workflows` folder contains a `resync-build.yml` workflow that runs weekly to keep workflow files in sync with the latest Pulumi provider CI standards. It opens a pull request for review before merging.

Set `PULUMI_PROVIDER_AUTOMATION_TOKEN` to a token with `pull-requests: write` permissions so that acceptance tests run on these automated pull requests.

## Manual Mappings

If automatic token mapping is insufficient, add resource and data source mappings manually in `provider/resources.go`:

```go
// Resources — snake_case HCL name → UpperCamelCase Pulumi token
"komodor_something":      {Tok: tfbridge.MakeResource(mainPkg, mainMod, "Something")},
"komodor_something_else": {Tok: tfbridge.MakeResource(mainPkg, mainMod, "SomethingElse")},

// Data sources — note the 'get' prefix
"komodor_something":      {Tok: tfbridge.MakeDataSource(mainPkg, mainMod, "getSomething")},
```

After any change to `resources.go`, re-run `make generate` to regenerate the schema and SDKs.

## Customise Provider Configuration

Override upstream provider configuration defaults in `provider/resources.go` as needed, for example to rename environment variables:

```go
"apikey": {
    Default: &tfbridge.DefaultInfo{
        EnvVars: []string{"KOMODOR_API_KEY"},
    },
},
```
